# 局域网 Web 编辑与协作 MVP 规划

> **状态：未实现。** 本文档用于记录后续可实施的最小可行版本（MVP）方向，避免方案讨论丢失。

## 目标

将该能力的第一阶段定义为：

**局域网 Web 辅助编辑 + 单写多读协作**

MVP 只解决以下问题：

1. 在同一局域网内通过浏览器访问项目
2. 支持少量低风险、结构化内容的远程修改
3. 支持多人同时查看，但同一时间仅允许一个写入者

## 明确不做的内容

以下内容不属于 MVP：

- 多人同时编辑同一份逻辑数据
- Web 端拖拽式 Block 编辑器
- 公网访问 / 穿透
- 自动冲突合并
- 复杂资源的在线编辑（如图片、音频）

## 建议的可写范围

第一批建议允许修改的内容：

- 项目基础信息
  - 项目名
  - 包名
  - 版本名 / 版本号
- 字符串资源
  - `values/strings.xml`
  - `values-zh/strings.xml`
- 结构化配置
  - Manifest 注入类内容
  - ProGuard / 项目设置类文本配置
- 命名类操作
  - 变量 / 列表重命名
  - MoreBlock 名称等可校验文本项

第一批建议只读的内容：

- 逻辑树 / 事件树
- 页面与组件结构
- Block 级逻辑内容

## 协作模型

MVP 采用：

**单写多读**

角色建议：

- 访客：只读
- 编辑者：唯一写入者
- 房主：可授予/回收写锁

写锁规则建议：

- 默认只读
- 编辑需要显式申请或授权
- 同一时间只能有一个写锁持有者
- 写锁超时自动释放
- 房主可强制回收

## Android 侧模块划分

### 1. `lan_entry`

负责局域网入口与开关：

- 开启/关闭服务
- 显示 IP、端口、PIN
- 控制默认关闭策略

### 2. `lan_auth_session`

负责认证与会话：

- PIN 校验
- 会话 token
- 超时处理
- 角色区分

### 3. `lan_locking`

负责单写多读控制：

- 写锁申请/释放
- 锁超时回收
- 房主强制回收
- 当前编辑者状态

### 4. `project_query_service`

负责只读查询：

- 项目列表
- 项目结构树
- 资源与配置读取
- 逻辑摘要只读输出

### 5. `project_command_service`

负责所有写操作：

- 修改项目基础信息
- 修改字符串资源
- 修改配置项
- 修改命名类内容
- 调用统一保存

### 6. `project_validation`

负责数据校验：

- 命名合法性
- 重名检查
- 包名规则
- 资源键合法性
- 当前状态是否允许写入

### 7. `project_persistence`

负责持久化与刷新：

- 保存项目数据
- 刷新缓存
- 向前端推送变更结果

### 8. `lan_api`

负责 API 暴露：

- HTTP 路由
- 请求/响应封装
- 调用 query / command service

### 9. `collab_notes`（可选）

负责轻量讨论：

- 项目评论
- 节点备注
- 简单讨论记录

建议使用 sidecar 数据存储，避免污染主项目数据结构。

## Web 侧模块划分

建议拆分为：

- `web_shell`
  - 登录、路由、会话状态
- `project_explorer`
  - 项目树 / 页面树 / 资源树
- `detail_viewer`
  - 只读详情展示
- `form_editors`
  - 字符串 / 配置 / 命名项编辑表单
- `lock_status_panel`
  - 写锁状态、申请与释放
- `notes_panel`
  - 评论与讨论

## 关键边界

必须遵守以下边界：

1. Web 端不能直接写项目文件
2. API 层不能直接依赖 `LogicEditorActivity` 等 UI 状态
3. 所有写操作必须先过 `project_validation`
4. 所有写操作必须通过 `project_command_service`

## 推荐实施顺序

### 阶段 1：只读链路

先完成：

- `lan_entry`
- `lan_auth_session`
- `lan_api`
- `project_query_service`

结果：浏览器可连接并只读查看项目。

### 阶段 2：单写可控

再完成：

- `lan_locking`
- `project_validation`
- `project_command_service`
- `project_persistence`

结果：可修改低风险结构化内容。

### 阶段 3：协作体验

最后补充：

- 锁状态提示
- 评论/备注
- 状态推送
- 更细粒度的错误反馈

## 成功标准

满足以下条件即可认为 MVP 有价值：

- 局域网连接稳定
- 单写多读模型可控
- 已支持的编辑内容确实能被用户高频使用
- Android 本地编辑器不会因 Web 修改产生状态错乱

## 后续扩展方向

后续如需演进，可考虑：

- 扩大可写范围
- 局部模块锁
- 差异预览
- 更强的讨论/审阅能力
- 最终再评估多人实时编辑与冲突合并
